## Deployment model — Docker

The gateway runs in a **local Docker container** mounted on the project directory. It is not a remote server — it runs on the developer's machine.

```yaml
# docker-compose.yml (in the project root)
services:
  gateway:
    image: ai-gateway:latest
    restart: always
    ports:
      - "127.0.0.1:8765:8765"   # never 0.0.0.0 — localhost only
    volumes:
      - .:/workspace:ro              # source read-only
      - ./.gateway:/workspace/.gateway  # .gateway in read-write
      - ollama:/root/.ollama
    environment:
      - WORKSPACE=/workspace
      # ANTHROPIC_API_KEY is never in environment — see secrets
    secrets:
      - anthropic_key

secrets:
  anthropic_key:
    file: .kiri/upstream.key   # mounted at /run/secrets/anthropic_key

volumes:
  ollama:
```

`restart: always` ensures the container restarts automatically on system reboot.

**Why not an env var for the key:** `docker inspect` and `docker exec env` expose environment variables in plain text. The Docker secret is mounted as a file at `/run/secrets/` and does not appear in either. See [ADR-003](../adr/ADR-003-docker-secrets.md).

## API key management — bypass prevention

The gateway adopts a **dual-key model** to prevent a developer from bypassing the proxy by calling Anthropic directly.

```
┌─────────────────────────────────────────────────────────┐
│  Developer                                              │
│  ANTHROPIC_API_KEY=kr-a1b2c3d4   ← kiri key         │
│  ANTHROPIC_BASE_URL=localhost:8765                      │
└───────────────────────────┬─────────────────────────────┘
                            │ calls with kr-a1b2c3d4
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Docker Container                                       │
│                                                         │
│  1. Validates kr-a1b2c3d4 → authorized developer        │
│  2. Runs Filter Pipeline                                │
│  3. If PASS: forwards with sk-ant-xxx (real key)        │
│                                                         │
│  ANTHROPIC_API_KEY=sk-ant-xxx  ← real key, internal     │
└───────────────────────────┬─────────────────────────────┘
                            │ calls with sk-ant-xxx
                            ▼
                    api.anthropic.com
```

**What happens if the developer tries to bypass:**
```bash
# removes the base URL and calls Anthropic directly
unset ANTHROPIC_BASE_URL
claude "refactor RiskScorer"
→ HTTP 401 — kr-a1b2c3d4 is not a valid Anthropic key
```

The real key (`sk-ant-xxx`) never leaves the container. The developer does not know it.

**Gateway key generation:**
- `kr-xxx` keys are generated by the container on first startup
- One key per developer, nominative — the audit log knows who did what
- Revoking a developer = invalidating their kiri key

**Note:** network-level bypass (firewall, transparent HTTPS proxy) is a more robust enforcement solution, but requires controlled network infrastructure.

## Sharing among developers — via git

The fundamental separation is between **configuration** (what to protect) and **index** (how it is encoded):

```
.kiri/
  secrets         ← COMMITTED to the repo — travels with the code
  config.yaml     ← COMMITTED to the repo
  index/          ← NOT committed — rebuilt locally by each dev
```

### Format of `secrets`

Supports two types of entries that can be freely combined:

```
# path — protects the contents of a file or folder
src/engine/risk_scorer.py
src/billing/

# symbols — protects a name regardless of where it lives in the code
@symbol RiskScorer
@symbol sliding_window_dedup
@symbol DataFlowEngine
```

**Paths** are fragile to refactoring — if the file is moved or split, protection breaks.
**Symbols** travel with the code — they survive renames, moves, and file splits.

The combination of both is the most robust strategy:
- paths for semantic protection (L1 similarity on content)
- symbols for immediate protection (L2, text matching, no DB required)

### Sharing flow

```
Dev A:
  "protect src/engine/risk_scorer.py"
  → kiri add → updates .kiri/secrets (path + extracted symbols)
  → git commit .kiri/secrets → git push

Dev B (git pull):
  → .kiri/secrets updated
  → L2 (symbols) active immediately — no DB required
  → Watcher detects the change → reindex L1 in background
  → full semantic protection as soon as reindex finishes
```

### Protection after refactoring

```
Before: secrets → src/engine/risk_scorer.py
                  @symbol RiskScorer

Refactoring: risk_scorer.py → scorer.py + window_dedup.py

Watcher detects that src/engine/risk_scorer.py no longer exists
→ notifies via Claude Code:
  "⚠️ Protected file not found: src/engine/risk_scorer.py
   Do you want to update the protection with the new files?"

→ @symbol RiskScorer continues to protect in L2
  even before the developer updates the paths
```

**To add to the project's `.gitignore`:**
```
.kiri/index/
```

## Components

### 1. Claude Code Skill — developer interface

The developer never uses direct commands. They interact in natural language inside Claude Code:

> "protect this file" → `kiri add <file>`
> "remove protection" → `kiri rm <file>`
> "what is protected?" → `kiri status`
> "why was it blocked?" → `kiri inspect <last_prompt>`

The skill is a translation layer: it receives natural language, calls the gateway CLI commands inside the container, and returns a readable response in chat.

**Activation:** installed automatically in the Claude Code environment at install time. The developer does not configure it.

---

### 2. Internal CLI (`kiri *`)

Runs inside the container. Used by the skill and the Watcher — the developer never sees it directly.

```
cli/
  commands/
    add.py        → kiri add
    remove.py     → kiri rm
    status.py     → kiri status
    inspect.py    → kiri inspect
    reindex.py    → gateway reindex
```

**Stack:** Python + Typer

---

### 3. Indexer + File Watcher

The Indexer reads source files and produces vectors and symbols — never raw code in the DB.

**Indexing pipeline:**

```
Source file (mounted via Docker volume)
    │
    ▼
Chunker (regex on indentation — per function / per class)
    │
    ├──► Embedding generator
    │      sentence-transformers/all-MiniLM-L6-v2 (offline, ~80MB)
    │      → float[] vectors
    │
    └──► Symbol extractor
           Ollama qwen2.5:3b (offline)
           → symbol list: ["RiskScorer", "sliding_window_dedup"]
    │
    ▼
Store (vectors + symbols only, never source code)
```

The **File Watcher** continuously monitors `.kiri/secrets`:
- If the file changes (git pull, kiri add/rm) → triggers automatic reindex
- For `@glob` rules, a background rescan thread (60 s interval) re-expands each pattern and indexes new files / purges disappeared files without requiring a secrets change
- No manual intervention, no git hooks to configure

---

### 4. Store

```
.kiri/                          (mounted from Docker volume)
  secrets            ← git ✅  paths and @symbols, one per line
  config.yaml        ← git ✅  thresholds, behavior, port
  index/             ← git ❌  rebuilt locally
    vectors.db       ← ChromaDB embedded (L1 similarity)
    symbols.json     ← symbols from secrets + extracted from files (L2)
    meta.json        ← { "path": { "indexed_at": "...", "hash": "..." } }
```

`symbols.json` is the merge of:
- symbols explicitly declared in `secrets` (`@symbol RiskScorer`)
- symbols automatically extracted by Ollama during reindex of file paths

It is rebuildable, but `@symbol` entries in `secrets` ensure critical symbols are
always available even without a DB.

**Stack:** ChromaDB embedded for vectors. JSON for symbols and metadata.

---

### 5. Filter Pipeline

Used by both the HTTP Proxy (runtime) and `kiri inspect` (dry-run).

```python
def filter(prompt: str) -> FilterResult:

    # L1 — Similarity Gate
    scores = vector_store.query(embed(prompt), top_k=5)
    if max(scores) >= config.hard_block_threshold:
        return BLOCK                    # high certainty, skip L2/L3
    if max(scores) < config.similarity_threshold:
        return PASS                     # below threshold, skip L2/L3

    # L2 — Symbol Filter (grace zone)
    found_symbols = symbol_store.scan(prompt)
    if found_symbols:
        return BLOCK

    # L3 — LLM Classifier (grace zone, no symbols found)
    verdict = ollama_classifier.classify(prompt, matched_tags)
    return BLOCK if verdict == "extract_ip" else PASS
```

**Default thresholds (`config.yaml`):**
```yaml
similarity_threshold: 0.75    # below: direct PASS
hard_block_threshold: 0.90    # above: direct BLOCK
action: block                 # block | sanitize
```

---

### 6. HTTP Proxy

Listens on `localhost:8765`. Exposes the Anthropic (and OpenAI, and Ollama) protocol to developer tools.

**Exposed endpoints:**
| Endpoint | Protocol | Used by |
|---|---|---|
| `POST /v1/messages` | Anthropic | Claude Code |
| `GET /v1/models` | Anthropic | Claude Code |
| `POST /v1/chat/completions` | OpenAI | Cursor, Copilot |
| `POST /api/chat` | Ollama | tools pointing to local Ollama |

**BLOCK error format** (Anthropic-shaped, not HTTP 400):
```json
{
  "type": "error",
  "error": {
    "type": "permission_error",
    "message": "Prompt contains protected IP: RiskScorer (src/engine/risk_scorer.py)"
  }
}
```

For SSE streaming:
```
event: error
data: {"type":"error","error":{"type":"permission_error","message":"..."}}
```

**Fail-open:** if the container is unreachable, calls pass directly to upstream. Work is never blocked.

**Stack:** FastAPI + async `httpx`.

## Data flow — no-code-in-cloud guarantee

```
What stays on-prem (in the container):   What can leave:
────────────────────────────────────────   ─────────────────────────────
Source code ✅                             float[] vectors (non-invertible)
Extracted symbols ✅                       User prompt (if PASS)
Ollama models ✅                           Remote LLM response
Vector index ✅
Config and policies ✅
```

## Technology stack

| Component | Technology | Reason |
|---|---|---|
| Deployment | Docker + docker-compose | Cross-platform, automatic restart |
| HTTP Proxy | FastAPI + httpx | Multi-protocol, native SSE streaming |
| Embedding (L1) | sentence-transformers | Offline, ~80MB, good semantic quality |
| Vector store | ChromaDB embedded | Zero server, embedded in project dir |
| Symbol extraction (L2) | Ollama `qwen2.5:3b` | On-prem, offline |
| LLM Classifier (L3) | Ollama (same process) | On-prem, grace zone only |
| File Watcher | watchdog (Python) | Monitors `.kiri/secrets`, triggers reindex |
| Config | YAML committed to repo | Shared across developers via git |

| Tool | Notes |
|---|---|
| Docker Desktop | Only dependency to install on the host system |

---

## Enforcement model limitations

The dual-key model prevents direct API bypass, but does not prevent
a developer with Docker access from inspecting the container to find the upstream key.
More robust enforcement would require network-level control (firewall, transparent HTTPS proxy)
outside the scope of the gateway itself.
