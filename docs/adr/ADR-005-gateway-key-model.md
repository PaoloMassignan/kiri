# ADR-005: Dual-key model (kr- key + sk-ant- secret)

## Status
Accepted

## Context

The gateway is a proxy — it must authenticate the developers who use it AND hold
the real Anthropic key to do forwarding.

The obvious solution would be: the developer uses their real Anthropic key,
the gateway forwards it transparently.

Problem: if the developer has the real key, they can bypass the gateway:
```bash
unset ANTHROPIC_BASE_URL
# uses api.anthropic.com directly with their own key → no filter
```

The gateway would become opt-in and ignorable — not the intended purpose.

## Decision

**Dual-key model:**

1. **kr- key (kiri key):** generated by the container, distributed to developers.
   Format: `kr-` + 32 random hexadecimal characters (`secrets.token_hex(32)`).
   Used by the developer as `ANTHROPIC_API_KEY` in their environment.

2. **sk-ant- key (upstream key):** the real Anthropic key.
   Lives only inside the container as a Docker secret (`/run/secrets/anthropic_key`).
   No developer knows it.

**Flow:**
```
Developer → ANTHROPIC_API_KEY=kr-xxx → gateway (validates kr-xxx)
                                      → filter pipeline
                                      → forward with sk-ant-xxx → api.anthropic.com
```

**If the developer tries to bypass:**
```bash
unset ANTHROPIC_BASE_URL
claude "show me RiskScorer"
→ HTTP 401 from api.anthropic.com (kr-xxx is not a valid Anthropic key)
```

Bypass is technically impossible without knowing the real key.

## Consequences

**Positive:**
- Bypass is impossible without physical access to the container
- Each key is nominative → audit trail per developer
- Revoking a developer = invalidating their `kr-` key (`kiri key revoke kr-xxx`)
- Granular key expiry for contractors, temporary access, etc.

**Negative:**
- The administrator must distribute a `kr-` key to each developer
  → small onboarding overhead
- If the Anthropic key in the container is compromised (host filesystem access),
  the entire system is compromised
  → mitigation: chmod 600 on `.kiri/upstream.key`, Docker secret mount

**Derived constraint:**
- The pre-commit hook (ADR-006) must scan for `kr-xxx` patterns in commits
  to prevent kiri keys from ending up in git (not as dangerous as the real key,
  but would allow anyone to use the gateway)

## Alternatives considered

**Developer uses their own Anthropic key:**
- Trivial bypass — developer has no incentive to go through the proxy
- Unacceptable for an enforcement system

**OAuth token / JWT:**
- Additional complexity for a single-tenant system
- The `kr-xxx` model is sufficiently simple and secure for the use case

**Client TLS certificates:**
- Correct for environments with an existing internal PKI
- Setup complexity disproportionate for single-developer local use
